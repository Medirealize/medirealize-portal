# 開発日誌 〜葛藤ログ〜

## 2026-05-08

### タイトル
**「使えるか」と「使いたいか」の差を、6点で埋めた日**

### 本日の葛藤（エピソード）
KiDuKi の入力画面は、機能としては動いていた。でも「動く」と「使いたい」は別物だということを、実際に触ってみて改めて感じた。  
診察前に緊張している患者さんが使う道具として、「操作に迷わせない」「進んでいる感覚を渡す」「信頼を途切れさせない」の3つが足りていなかった。

変更ファイルは `InputScreen.tsx` 1本だけ。でも今日の実装には、ずっと先送りにしていた問いへの答えが詰まっている。

**1. プログレスバー**  
「今どこにいるか」が見えないと、人は不安になる。医療現場でも、患者が「あとどのくらいで診てもらえるか」を知りたがる理由と同じだ。  
入力 → 整理中 → 問い1〜3 → まとめ作成中 → 完了 の6ステップをバーで可視化し、「あと○ステップ」のテキストを添えた。  
`progressMap` という `Record<Step, {label, current}>` 型のマップで管理し、進行率は `(current / 6) * 100` の単純計算。複雑さより伝わる速さを優先した。

**2. 最短モード（「最短で作成」ボタン）**  
「急いでいるとき」と「まず結果を見てから考えたいとき」に、3問に答える手順を省けるようにした。  
内部的には各問いの選択肢の先頭を自動選択して一括処理。`SubmitMode = "normal" | "quick"` の型で分岐させ、通常フローと同じ `doReframe()` を経由するので出力品質は落ちない。  
エラー時のリトライも `lastSubmitMode` に最後のモードを記憶し、同じモードで再送する。

**3. 例文挿入ボタン**  
「何を書けばいいか分からない」で止まる人への一手。`setInput(EXAMPLE_TEXT)` を呼ぶだけの実装だけれど、最初の一文が浮かばなくて諦めてしまうユーザーへの距離を縮める効果は大きい。

**4. プライバシー注釈のアコーディオン化**  
「保存されない」という一文では足りなかった。「本当に？」と思う人に向けて、詳しい説明をトグルで開けるようにした。  
離脱の原因がプライバシーへの不安である場合、説明を押しつけるより「知りたい人だけ読める設計」のほうが信頼を作る。

**5. 文字サイズ切替**  
対象ユーザーの主な年齢層は40〜70代だ。「標準」「文字大」のトグルを右上に設置し、`FontMode` に応じて `mainTextClass / subTextClass / noteTextClass` を切り替える。  
設定を記憶させていないのは意図的。毎回設定する手間より「いつでも切り替えられる」という安心感の設計を選んだ。

**6. エラー時の再送導線**  
通信エラーで止まったとき、従来は「また最初から」という絶望だった。エラーメッセージの下に「もう一度送信する」ボタンを出し、`retryLastSubmit()` で同じモードのまま再送できるようにした。  
入力が5文字未満の場合は非表示にして、バリデーション済みのケースだけに絞っている。

### 本日の「共犯者特典」
今日の知恵は、**「動く」を「使える」に変えるための6つの観点**をセットで持っておくこと。  
プログレス・最短経路・入力補助・プライバシー説明・アクセシビリティ・エラー回復——この6点は、どのフォーム系UIにもそのまま移植できる診断リストになる。

実装上の要点（そのまま使える）:

1. 進行バーは「ステップ数を固定して比率で計算」する方が、条件分岐より安定する
2. `SubmitMode` 型で処理を分岐させると、エラー時のリトライ追加が容易になる
3. プライバシー説明は「全文常時表示」より「展開可能」の方が、不安な人への伝わり方が上がる
4. 文字サイズの切替は CSS クラスを変数に束ねておくと、変更箇所を最小にできる

1ファイルの修正でも、使いやすさへの投資は積み重なる。

## 2026-05-08

### タイトル
**計測・PWA・安定性──「届くプロダクト」へ引き上げた日**

### 本日の葛藤（エピソード）
ほんね。は動いていた。でも「動く」と「信頼できる」は別の話だった。  
今日はその差を埋めることに集中した。

機能を増やし続ける前に、「計れない問題は直せない」という事実と正面から向き合った。  
`/api/analytics` エンドポイントを新設してクライアント行動を計測できる基盤を置き、チャット API・深掘り API にタイムアウトとサーバーログを追加した。遅延や障害がどこで起きているかをログで把握できる状態にする——これが今日の最優先事項だった。

合わせて Service Worker と Web App Manifest を追加し、スマートフォンのホーム画面にインストールできるようにした（PWA 対応）。  
ブラウザのタブではなく「アプリ」として手元に置いてもらえる状態になった。

ナビゲーションのレース条件は、連続タップで画面が誤遷移するバグだった。  
`navEpoch` を導入してキャンセル処理を実装し、タップが速くても正しいページへ進むようにした。`onClick` へのイベントオブジェクト誤渡しも合わせて解消した。

Playwright によるスモークテスト（`e2e/smoke.spec.ts`）も整備した。AI オフ時のフローを自動検証できるようになり、デプロイ後の無言の壊れを検知できる体制になった。

### 本日の「共犯者特典」
今日の知恵は、**「計測・回復・テスト」を同じリリースでセットにする**こと。  
計測なしでは改善の根拠が取れない。エラー回復なしでは信頼を失う。テストなしではデプロイが怖い。3点を揃えることで、次のフェーズへ踏み込む足場ができる。

実装上の要点（そのまま使える）:

1. analytics エンドポイントは POST のみ・認証不要の軽量設計にしておく
2. API タイムアウトは `Promise.race` で制御し、エラーメッセージをユーザー向けとサーバーログで分ける
3. PWA の Service Worker はキャッシュ戦略を明示しないと本番で予期しない挙動が起きる
4. ナビゲーションのレース条件は `epoch` 番号を state で管理し、非同期完了時に最新かどうか確認してから遷移する

## 2026-04-14

### タイトル
**動くコードを、触れるコードへ整理した日**

### 本日の葛藤（エピソード）
ほんね。の機能追加が続いた `app/page.tsx` と API ルートが、少しずつ重くなっていた。  
書いた自分は分かる。でも1週間後の自分には分からない。  
今日は「動く」から「触れる」コードへの整理に集中した。

- パーソナリティタイプの定義とロジックを `lib/personality.ts` へ分離
- インサイト生成のヘルパーを `lib/insights.ts` へ分離
- 共有定数を `lib/constants.ts` に集約
- API ルートの `as any` を型付きの `OpenAIChatCompletion` に置き換え
- `deep-questions` の fallback で空文字列を `split` していたバグを修正（問いが空になるケースを解消）
- 使われていない `app/api/ai/route.ts` を削除

コードを動かすのは最初の1回だけど、読むのは何度も繰り返す。  
今日の整理は、次の機能追加を速くするための先行投資だった。

### 本日の「共犯者特典」
今日の知恵は、**「分離・型付け・削除」を同じタイミングで行うリファクタリングの三点セット**。  
ロジック分離だけすると型が曖昧なまま移動する。型付けだけすると不要コードが残る。削除だけすると依存関係を見落とす。3点セットで行うと、整理後に「なぜここに置いたか」が明確になる。

## 2026-04-14

### タイトル
**深掘り問いを「多様に、短く」に調整した日**

### 本日の葛藤（エピソード）
ほんね。の深掘り問いが、同じ切り口で並ぶことがあった。  
3つ提示しているのに、差が1つしかない——という体験だ。

生成プロンプトに「それぞれ異なる視点から」の制約を追加し、同じ切り口が連続しないようにした。  
あわせて、モバイルで2行に収まりやすい文字数ガイドラインへ緩和した。  
スマートフォンで問いが3行にわたると、画面を圧迫して選びにくくなる。「短く伝わる」が「詳しく伝える」より価値を持つシーンがある。

生成ごとのバリエーションを確認するため、サーバーログを一時追加した（後続リリースで整理予定）。

### 本日の「共犯者特典」
今日の知恵は、**AI 生成の問いには「視点の被り」に対して明示的に制約をかける**こと。  
「3問提示して」だけでは同じ視点が並ぶことがある。「それぞれ異なる視点から」をプロンプトに追記するだけで、出力の多様性は大きく変わる。

### 本日の葛藤（エピソード）
開発日誌は書いているのに、公開側の日付や導線が追いつかないと、読者には「止まって見える」。  
このズレは、実装の問題というより、信頼の見え方の問題だった。

- 最終更新表示（`DEVLOG_VIEWER_LAST_UPDATED`）を `2026-04-13` に更新  

これで、長文を読む人と一覧で追う人が、同じ時間軸で変化を追えるようになった。

### 本日の「共犯者特典」
今日の知恵は、**更新作業を「文章」と「日付同期」に分けず、同時に完了させる**こと。  
日付の整合性は地味だけれど、継続プロダクトでは一番効く信頼資産になる。  

## 2026-04-09

### タイトル
**分割された美しさを、運用できる一枚へ戻した夜**

### 本日の葛藤（エピソード）
今日は「UIの正しさ」と「運用の現実」のせめぎ合いだった。  
v0で作った見た目は確かに洗練されている。でも、コンポーネントが細かく分かれすぎると、微修正のたびに触る場所が増えて、意思決定が遅くなる。  

現場で例えるなら、器械は最新でも、配置が散らばっていて手が止まる手術室に近い。  
「再利用性」は正義だけれど、今このフェーズでは「変更速度」も同じくらい正義だった。  

そこで今回は、ランディングのUIを `app/page.tsx` に一旦集約。  
ただし、見た目だけ寄せるのではなく、内容は既存の正データを優先した。  

- ヘッダー〜フッターまでのLP UIを `page.tsx` にインライン統合  
- **4つのアプリ導線**は既存定義（`FEATURE_DEFINITIONS`）を維持  
- **開発日誌の価格**は `月額100円` 表記を既存定数から参照  
- 既存の法務・問い合わせ導線の整合性を崩さないよう調整  

結果として、デザイン再現と運用負荷のバランスが取れた。  
「分ける」ことより先に、「今すぐ直せる」ことを優先した一日だった。

### 本日の「共犯者特典」
今日の知恵は、**“設計の正しさ”より“変更の速さ”が勝つフェーズを見誤らない**こと。  
特に立ち上げ期のLPは、抽象化よりも「一箇所で全体を触れる状態」が強い。  

実装上の要点（そのまま使える）:

1. UI統合時も文言・価格・リンクは**定数参照**を維持する  
2. 見た目の再現と業務要件（法務/導線）を同時に満たす  
3. まずは1ファイルで速度を取り、安定後に再分割する  
4. 「触るファイル数」を減らしてレビュー負荷を下げる  

この4点を守るだけで、デザイン刷新の失敗確率は大きく下がる。  
明日は、統合後のUIをスマホ実機幅で再点検し、必要なら段階的に再コンポーネント化します。

## 2026-03-27

### タイトル
**決済導線と診察導線が、同じボタンで衝突した夜**

### 本日の葛藤（エピソード）
一番詰まったのは、技術じゃなくて「意味」だった。  
`開発日誌を閲覧する` を押した人がStripeに飛ぶ。仕様としては正しい導線に見えて、体験としては完全に逆。ここで一気に温度が下がった。  

医療現場で言えば、トリアージと診察室の入口を同じドアにしているようなものだ。  
「閲覧したい人」と「購読したい人」を同じ動線に置いた瞬間、ユーザーの頭の中ではノイズになる。  

正直、ここは少しイラついた。ボタンの文言を何回直しても、根本は導線設計だったから。  
最終的にやったことはシンプルで、でも重かった。  

- フッターCTAを**購読する / 閲覧する**に分離  
- `/dev-log` は**購読者クッキー必須**でガード  
- `/dev-log/unlock` でメール照合し、購読者だけ通す  
- Stripe決済は「購読」のみ、閲覧は「認証」のみに役割分離  

コードとしては小さな変更の積み重ねだけど、体感は大手術に近かった。  
「曖昧な1タップ」を消すだけで、サービス全体の信頼が一段上がる。  

### 本日の「共犯者特典」
今日の知恵は、**“支払い導線”と“権限導線”を絶対に同じUIにしない**という一点に集約される。  
これを守るだけで、医療DXプロダクトの離脱率と問い合わせ率は目に見えて下がる。  

実装上の要点（そのまま使える）:

1. `unlock` ページを必ず中継点にする  
2. `cookies()` で購読状態を判定し、未購読は即 `redirect`  
3. 価値コンテンツは `/dev-log/resources/*` に切り出し、同じガードを適用  
4. 法務ページ（特商法/規約）と決済説明文を必ず一致させる  

この4点を押さえるだけで、審査対応とUX改善を同時に前進できる。  
これを知っているだけで、あなたも医療×ITの最前線に立てる。  

明日は、Stripe webhook と購読状態の自動同期の山に登ります。
