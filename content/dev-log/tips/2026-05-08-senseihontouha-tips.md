# 2026-05-08 ほんね。 の実践Tips

## 今日の要点
- 「計測・エラー回復・テスト」は機能追加と同じタイミングで整えると、次のフェーズへの足場になる
- analytics エンドポイントは最初から軽量設計にしておかないと、後から分離するコストが大きくなる
- PWA の Service Worker はキャッシュ戦略を明示しないと本番で予期しない挙動が起きる

## そのまま使える手順
1. analytics エンドポイントは POST のみ・認証不要・レスポンス 200 固定の最小設計で始める
2. API タイムアウトは `Promise.race([fetch(...), timeout(ms)])` で制御し、タイムアウト時のエラーを別途ログに出す
3. Service Worker では `install` / `fetch` イベントを最低限ハンドルし、キャッシュ対象を明示的に絞る
4. ナビゲーションのレース条件は `const epoch = ++navEpoch.current` を非同期処理の先頭で取得し、完了時に `epoch === navEpoch.current` を確認してから遷移する
5. Playwright のスモークテストは AI 呼び出しをモックしてオフにし、UI フロー全体を確認する

## 一言メモ
動くプロダクトと信頼できるプロダクトの差は、計測・回復・テストの三点が揃っているかどうかで決まる。
